# Архитектура Linux [Draft]


В архитектуре Linux выделяются следующие уровни и компоненты:
1) Пользовательское пространство / User space
2) System Call Interface
3) Пространство ядра / Kernel space
4) Аппаратное обеспечение / Hardware
5) Другие компоненты

![](/data/images/png/linux/architecture.png)

### 🚪1. Пользовательское пространство (User space)

Верхний уровень, где работают:
- Ваши пользовательские приложения / Applications: браузер, игры, IDE.
- Утилиты / Utilities: ls, grep, bash, curl.
- Библиотеки (прослойки) / Libraries / libc, такие как: glibc, musl, pthread — оборачивают системные вызовы в удобные функции (printf → write)
- Процессы, Изолированные экземпляры программ / Processes: каждый процесс имеет свой PID, виртуальную память, файловые дескрипторы

>  !: Приложения НЕ МОГУТ обращаться к железу напрямую. Любая операция ввода-вывода или управления памятью требует перехода в Kernel Space.

### 🚪 2. System Call Inerface (Шлюз системных вызовов)

Граница между пользовательским и режимом ядра.

__Механизм перехода__:\
Инструкция процессора syscall (или int 0x80 на старых x86)

__Что происходит:__:
1. Переключение процессора в привилигированный режим (Ring 0);
2. Сохранение контекста текущего процесса;
3. Поиск номера вызова в таблице;
4. Выполнение функции ядра.

__Примеры вызовов__: `fork()`, `execve()`, `read()` `write()`, `mmap()`, `socket()`.

🔎 __Диагностика__: Чтобы увидеть системные вызовы в реальном времени, используйте утилиту `strace`:
```bash
strace -c ls /tmp  # покажет статистику вызовов для команды ls
```

### 🚪 3. Пространство ядра (Kernal Space)

Мозг операционной системы. Работает с полным доступам к ресурсам.

|    Подсистема     |        Функция |
|-----------|---------------------|
| Process Scheduler |  Решает какой процесс выполняется на CPU (CFS - Completely Fair Scheduler)    |
| Memory Manager (MMU)     |  Управляет виртуальной памятью, страницами, swap, защищает память процессов друг от друга  |
| Virtual File System (VFS)      | Абстракция над файловой системой (ext4, xfs, tmpfs, procfs) |
| Device Drivers      |  Код управления конкретным железом (диск, сетевая карта, USB)  |
| IPC      |  Механизм межпроцессного взаимодействия: pipe, очереди, сообщения, shared memory  |

> Модули ядра (Kernel Modules): Динамически загружаемые части ядра (.ko файлы), позволяющие добавлять драйверы или функции без перезагрузки.

### 🚪 4. Железо (Hardware)

Аппаратное обеспечение. Физический уровень:
- CPU: Процессор выполняет инструкции, генерирует прерывания.
- RAM: Хранит код и данные (физические адреса).
- Devices: Диски (NVMe/SATA), сетевые карты (NIC), GPU, USB-контроллеры.
- Interrupt Controller (APIC): Маршрутизирует аппаратные прерывания от устройств к ядру.

### 🚪 5. Другие компоненты архитектуры Linux

Эти компоненты важны для понимания работы Linux, часто не изображаются на схеме:

|    Компонент          |        Что делает |
|-------------------------|---------------------|
| Shell/Terminal Emualtor |  Интерфейс между пользователем и ядром. Сам shell - это приложение уровня пользователя / User Space, которое делает системные вызовы (`exec`, `fork`) для запуска команд   |
|    Virtualization Layer (Hypervisor)   |    Если Linux работает в виртуальной машине (KVM, VMware), между ядром и железом появляется слой виртуализации, эмулирующий устройства    |
|   Security Modules (LSM)    |   SELinux, AppArmor - перехватывают системные вызовы для проверки прав доступа (Mandatory Access Control)     |
|   /proc и /sys (procfs/sysfs)    |     Виртуальные файловые системы, через которые ядро экспортирует внутреннюю статистику в User Space (информацию о процессах, устройствах)   |
|  Context Switch Data     |   Данные, сохраняемые при переключении задач: регистры CPU, указатель стека, таблицы страниц памяти     |

Рассмотрим поток данных на примере выполнения команды:
```bash
cat file
```
Что происходит:
1. `User Space`: `cat` вызывает `fopen()` → `libc` вызывает `read()`.
2. `Syscall`: Выполняется инструкция `syscall` с номером `SYS_read`.
3. `Kernel Space`:
- ядро находит файл через VFS.
- Драйвер файловой системы читает метаданные с диска.
- Данные копируются из `Kernel Buffer` в `User Buffer`.
4. `Return`: Управление возвращается в `cat`, данные выводятся на экран (через `write()` в `/dev/tty`).

#### 🛠️ Как исследовать архитектуру на практике

```bash
# 1. Посмотреть системные вызовы процесса в реальном времени
strace -p <PID>

# 2. Изучить таблицу системных вызовов вашего ядра
ausyscall --dump          # Debian/Ubuntu
# или посмотреть в исходниках ядра: arch/x86/entry/syscalls/syscall_64.tbl

# 3. Посмотреть, какие модули ядра загружены
lsmod

# 4. Изучить структуру процессов
ps auxf                   # дерево процессов
top -H -p <PID>           # потоки внутри процесса

# 5. Посмотреть прерывания в реальном времени
watch -n 1 'cat /proc/interrupts'
```
